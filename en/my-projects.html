==== Update: 04/09/24 ====
As I have learned about new technologies, my usual approach has changed over time: I plan my projects with the Kanban board function in Github Projects.

Desired changes are created with the help of a self-created Github issue template. Within the template, the Gherkin syntax for Behavior-Driven Development (BDD) is used.

The issues are sorted by priority. I use the Scrum methodology and define the sprints for what should be done next in the near future. Issues with the highest priority are completed first.

Once it is clear what needs to be done, I use the Jest testing framework to create the tests that need to be fulfilled afterwards (Test-Driven Development, TDD).

After a sprint, I reflect on what could be done better. The next sprint is then planned and implemented.

QM Interface: Quipsy - Inspection Plan including Skip-Lot Procedure

Expansion of the interface between SAP and the external QM software Quipsy. This involved sending data from SAP to Quipsy (inspection plan including skip-lot procedure). And vice versa, data was sent from Quipsy to SAP (Quipsy database changes to the article inspection plan assignment).

General information about the project
  • Client: Leading manufacturer in the office chair, lounge furniture and automotive industry, > 2,000 employees
  • Project Goal: Transfer of the inspection plan including skip-lot procedure in SAP incoming goods inspections from SAP to Quipsy to relieve the quality management department. Special features: There is no Quipsy test system. Testing must therefore be carried out in the productive system. There is also no QM SAP consultant who is familiar with the QM module in SAP.
  • Number of Project Participants: ~ 8 Employees.
  • Time Period: 10/2023 - 12/2023
  • Industry / Area: Office chair, lounge furniture and automotive industry
  • Role: Lead Developer (SAP-side).
  • Language: German
  • Result: Successful transmission of the skip-lot procedure including the associated inspection plan. This significantly reduces the workload of the QM department. In addition, automatic updating of the article inspection plan assignment database table (SAP-side), which minimizes maintenance effort and errors. Possible errors are intercepted and written to an error log. As incorrect database changes may only be noticed too late, notifications are sent to the relevant employees when new entries are made in the error log.
  • Tech Stack (in relation to my role within the project):
    • ABAP (Procedural, OO)
    • ABAP Eclipse
    • Quipsy-Server (Task Scheduler, Log-, Ini-Files, ...)
    • SAP Database (SE11, SM30)
    • SAP Directory (AL11)
How I worked on this project

The special features of the project that make it much more difficult:

  • There is no Quipsy test system. This means that the data exchange between SAP and Quipsy had to be tested in the productive system. Attention had to be paid to the correctness of the data (purchase order, goods receipt). Also additional organizational effort, as several departments including Quipsy consultants are involved.
  • No in-house SAP QM consultant available. As it was not completely clear how to create and control skip lots or inspection plans in SAP (including the „side effects“), the inspection plans (article inspection plan assignment) had to be exported from Quipsy to SAP and kept up to date.
  • Lack of knowledge about the interface. The interface was developed by the SAP service provider NTT DATA. Only documentation was available. None of the people responsible at the time were still employed. As a result, I had to teach myself everything (contexts, code base, ...).

In order to check whether the data (test plan incl. skip lot) was being pulled correctly into SAP, it was necessary to run through the entire goods receipt process for each test. This means that a purchase order has to be created, an inbound delivery created, distributed and maintained in EWM.

An external Quipsy consultant was required for Quipsy-side adaptations.

Initially, only the skip-lot procedure (skip-lot button in the material master) was to be imported from SAP into Quipsy. However, as problems arose, the associated inspection plan also had to be imported into Quipsy.

As mentioned above, it was not clear how complex or feasible the introduction of inspection plans in SAP would be. For this reason, a special SAP database table was created for the article inspection plan assignment in SE11. This was initially filled via file upload.

An SM30 table and associated user documentation were also created so that the Quipsy key user can manually update the SAP database table when changes are made. This was necessary because the automatic export of the article inspection plan allocation database table in Quipsy to SAP had not yet been implemented and development had also been delayed.

Once the export of the skip lot including the inspection plan had been successfully implemented and tested, the next step was to automatically export the article inspection plan assignment database table from Quipsy to SAP in order to be able to determine the correct inspection plans in the future. The interface now runs fully automatically.

According to Quipsy, it is not possible to export individual database changes. The complete database table always had to be transferred to SAP. This Txt file had to be prepared in the ABAP code (e.g. remove separators or check whether the material number in Quipsy even exists in SAP).

To make troubleshooting as easy as possible, an error log with the incorrect database changes has also been implemented, which uploads a text file to the SAP directory.

To ensure that these errors do not rot away in a dark room, but are rectified immediately, a notification is sent to the relevant employees when a new entry is made in the error log.

Finally, the code was refactored and the existing documentation was updated and made available to the in-house SAP and QM departments.

Why I implemented the project in this way

Much has already been described in the previous section. Perhaps there were better solutions. I also had to use the existing code base and interface.

To make it easier for the developers after me, I documented the entire incoming goods receipt process, which makes testing easier. I also carried out code refactoring and always recorded why something had to be implemented exactly the way it was.

My 3 most important points that I have learned from this project
  1. Goods receipt process: Learned a lot about the functional side. Interrelationships became much clearer. Also got to know better how to create and maintain orders and deliveries as well as the Quipsy QM software.
  2. It is very interesting to see how a leading SAP service provider implements and documents an interface.
  3. Technically, I also learned a lot: How do I create a database table (SE11 as well as SM30)? What is an SAP directory? How do I develop the upload of files (via SAP GUI as well as AL11)? How does the Quipsy interface work? How is the Quipsy server structured? And so on.

Rollout: Form Offer

Introduction of a new SAP form: Offer (Module: SD).

General information about the project
  • Client: Leading manufacturer in the office chair, lounge furniture and automotive industry, > 2,000 employees
  • Project Goal: Introduction of a new SAP form (Offer), which is then to be used by Sales. Used for process optimization.
  • Number of Project Participants: 4 Employees (SAP Developer, SAP Consultant, Sales: Head of Department and Tester)
  • Time Period: 05/2024 - 06/2024
  • Industry / Area: Office chair, lounge furniture and automotive industry
  • Role: SAP Developer
  • Languages: German, English
  • Result: Successful introduction of the offer form, which is used by the entire sales department. Simplification, acceleration and error minimization in the creation of offers.
  • Tech Stack:
    • ABAP (Procedural, OO)
    • XML
    • ABAP Eclipse
    • SAP Form Technology Adobe Forms
    • SAP Standard Enhancement (Customizing the Output Type, Incompletion Log, ...)
How I worked on this project

It all started with the requirements analysis. My predecessors had already developed an incomplete offer form. The data came from the same print program as the Bock order confirmation form.

Sales used the two forms to define their desired customized offer form. I discussed this step by step with the sales department and recorded it in development documentation. The development can begin.

The previous offer form was a copy of the SAP standard offer form and was modified slightly. It contained bugs, unsightly as well as dead code. Bug fixing and code refactoring were therefore at the top of my to-do list.

I then developed new functionalities. It was a cycle of the following steps:

  1. Implement new features
  2. Testing (Myself)
  3. Discuss with Sales
  4. Testing (Sales)
  5. Obtain feedback. Document
  6. If necessary: Bugfixing

The code was refactored at the end of the project. The development documentation was adapted, formatted and made available to the project participants and the entire SAP team.

Why I implemented the project in this way

Before I put a lot of time into the development, I thought it was important to record the requirements together in writing and clearly. This had several advantages such as:

  1. Reduce subsequent changes to a minimum (time saving)
  2. Facilitates the handover to other developers as well as the mental switching between several SAP projects
  3. Due to the clarity, the status of the many changes can be better discussed / identified (What is finished? What still needs optimization? What is missing?)

I also attached great importance to a cycle between implementation and feedback. In order to be able to recognize possible course changes at an early stage and to keep testing easier, as changes are divided into units or blocks.

My 3 most important points that I have learned from this project
  1. When analyzing requirements, divide a form (layout) into blocks (header left, header right, before item table, item table itself, after item table, ...). Gives the whole thing more structure and facilitates consultation with the department.
  2. Create a before-and-after overview of the requirements and changes and update it regularly. This also gives the whole thing more structure and facilitates consultation with the department.
  3. More knowledge about the SD module with regard to the offer process, header and item details.

Pricing Report: Dynamic Programming

A report on all price conditions in sales. The special feature: The selection screen, the database query and the ALV table are created dynamically (during runtime).

General information about the project
  • Client: Leading manufacturer in the office chair, lounge furniture and automotive industry, > 2,000 employees
  • Project Goal: Report on all possible pricing conditions (transaction VOK0). Each condition type has an access sequence with several accesses. Each access has a database table. Each database table has several fields / primary keys. Several different condition types can be queried at the same time.
  • Number of Project Participants: ~ 4 Employees (SAP Developer, SAP Consultant, 2 Tester)
  • Time Period: 06/2024 - 07/2024
  • Industry / Area: Office chair, lounge furniture and automotive industry
  • Role: SAP Developer
  • Language: German, English
  • Result: Report with which all possible price conditions and their condition records can be displayed. It scales beautifully, as new price conditions do not have to be programmed separately. Even if existing price conditions (incl. access sequence, accesses and fields) are deleted or changed, no adjustment is necessary. Possible errors during use are intercepted and users and developers are informed accordingly with a content-rich message.
  • Tech Stack:
    • ABAP (Procedural, OO)
    • ABAP Eclipse
    • ABAP-Unit-Testing
    • SAP Reports (ALV)
    • SAP Standard Enhancement (Customizing)
How I worked on this project

Once the requirements were clear, I initially had no idea how to implement the project. How do I write a report without having to create at least 50 selection screens, database queries and output tables?

That would be an incredible amount of work. And if new price conditions are added or changed or deleted, maintenance work is required.

When I drew a bird's eye view of the procedure in transaction VOK0 (Customizing pricing) on a sheet of paper, I had the golden idea: dynamic programming.

So far, I had only dealt with it superficially in the course „ABAP for advanced users“ of the TH Brandenburg (see About me page, section Certificates for more information). So I first had to read up extensively and try things out.

A lot of testing, documentation and code refactoring was carried out during development. Unit tests were also written.

Once the report was finished, the final tests were carried out and the code and documentation were refined. I then presented it.

Why I implemented the project in this way

As each price condition has a different access sequence including several accesses, each with a different database table + selection fields, manual, static programming is not possible.

In addition, there is also the requirement to be able to query several condition types at the same time. This in turn makes the program considerably more complex (e.g. only one field called „sales organization“ can be displayed dynamically in the selection screen, although it occurs several times in different databases. In addition, the sales organization must also be entered accordingly in each database table, if available).

That would be an incredible amount of writing work if the program were written statically. In addition, it would not scale well and would require a lot of maintenance when price conditions change.

As no one in the company had mastered dynamic programming, I placed a lot of emphasis on documentation and code refactoring. So that my colleagues could adapt the program as quickly and easily as possible - in case I was making a nice life for myself abroad at the time, for example ðŸ˜‰

Some error handling was carried out in the report. So that users and developers are not left in the dark. With dynamic programming, you also have to understand your craft, as there is no syntax check, for example, and the code is only defined at runtime.

I have also written unit tests so that various tests can be run automatically at the touch of a button. This not only saves a lot of testing effort. It also makes it much easier to expand the program. You can see directly whether the program is still running as intended. This benefits both my colleagues and myself!

My 3 most important points that I have learned from this project
  1. Dynamic programming: At the time of the project, nobody in the company was familiar with the technology. In other words, I was on my own. Through research and trial and error, I now know a lot more (dynamic selection screen, database query, output table, ...).
  2. Paper scribbles can be worth their weight in gold. By outlining the entire price condition process, I was quickly able to find the common denominator that made dynamic programming possible in the first place.
  3. Pricing: I was previously unfamiliar with pricing conditions and transaction VOK0 (Customizing pricing). I am very impressed by how much is behind it.

  • Project Goal: Changeover from previous ERP software Macola to SAP. Previously, SAP was only used in the plants in Germany and Hungary. Canada (> 150 employees) is now also to use SAP for the purpose of standardization. Advantages include simpler controlling and management of resources.
  • Number of Project Participants: ~ 40 - 60 Employees („Normal“ Canadian employees excluded, only executives or main persons responsible of the respective departments in Canada included)
  • Time Period: 10/2022 - 10/2023
  • Industry / Area: Office chair, lounge furniture and automotive industry
  • Role: Lead developer for forms (in-house). Also supported user & authorization management and various smaller tasks.
  • Language: German, English
  • Result (general): Successful SAP integration in Canada. The most important processes run smoothly. Everything go-live critical in all possible SAP modules (PP, MM, WM / EWM, SD, FI, CO, ABAP, BASIS) covered. Minor optimizations / training after go-live.
  • Result (in relation to my role): Successful development of more than 8 forms in various SAP modules (invoice, order confirmation, bill of lading, commercial invoice, customer complaint, ...), which are used throughout the Canadian plant. Including adaptation of existing as well as newly developed forms. Conversion of the DIN A4 paper format to letter or Canada-specific dimensions. Minor optimizations after go-live.
  • Tech Stack (in relation to my role):
    • ABAP (Procedural, OO)
    • XML
    • SAP Form Technology Adobe Forms
    • SAP Reports (ALV, Queries)
    • SAP Printer Administration
    • SAP Authorization & User Administration
    • SAP Standard Enhancements (Code as well as Customizing of Output Types and Print Control)
How I worked on this project

This was my first major SAP project. At the beginning of the rollout, I was an SAP novice. Although I had previously been involved in software development outside of SAP, I was still largely unfamiliar with SAP (both technically and functionally).

Our senior developer had given me a brief introduction to forms development - in general and in relation to the company's previous code base. He had previously only worked with the predecessor form technologies SAPscript and Smart Forms.

In order to understand the latest form technology Adobe Forms and other contexts, I familiarized myself with the topic with the help of the book „SAP Interactive Forms by Adobe“ from SAP Press.

Other topics such as customizing output types, SAP printer control and functional relationships / processes were taught to me by colleagues in my department.

The planning of developments took place with the respective SAP consultants and departments. My technical questions were clarified by our senior developer, Google or the SAP form book. Once the requirements had been clarified, the development work began.

I documented my developments, tests etc. in detail. The documentation was available to the entire project team. I carried out the tests myself during development and afterwards with the respective departments and SAP consultants.

Why I implemented the project in this way

I documented my developments in detail (both in Word and within the code) in order to simplify and speed up subsequent adjustments on my part or by my colleagues.

I attached great importance to clarity and why comments (e.g. Why was exactly this field used in transaction XYZ and not another?).

I also used object-oriented ABAP. Our previous forms were largely implemented procedurally (FORM routines, function modules, ...).

Object-oriented programming had great advantages over procedural programming, such as:

  • More intuitive code, as objects are imitated in the real world
  • Inheritance, which enables code reuse (e.g. for standardized headers and footers of forms)
  • ABAP OO has a stricter syntax check than procedural ABAP and therefore allows fewer bugs and nicer, more readable code.
  • Modularity. What belongs together stays together.
  • Due to the advantages mentioned above, the software is more scalable overall and maintenance/extension is more efficient. There are even more reasons why the software becomes more scalable and maintainable with ABAP OO.
My 3 most important points that I have learned from this project
  1. Project management: How do I plan and manage a project? What points need to be considered with regard to form development? How do I document a project?
  2. How do I deal with stress and responsibility? There was a lot of time pressure during development. Sleepless nights included.
  3. Form development: Functional (e.g. goods receipt, goods issue, production, order) as well as technical (output types, print program, interface, layout, printer, ...)
